# ConstraintAttachmentF1 metric migration

One source of truth: the before/after table below is regenerated by running
`uv run python -m src.evaluation.metrics --gold_dir datasets/archive/gold_human
--pred_dir logs/baseline_runs/run_1 --tier A --threshold <T>` with the two
revisions of `src/evaluation/metrics.py` (pre-fix at `main@9e358ef`, post-fix at
HEAD of `eval/issue-53-gold-dataset-expansion`).

## Root cause

`ConstraintAttachmentF1` was effectively unmeasured because of a **schema
mismatch**, not strict string matching. The metric read top-level
`gold_doc["constraints"]`, but the gold files store constraints **nested per
step** at `steps[i].constraints.{precondition, postcondition, guard,
acceptance_criteria, warning, exception}`. That key did not exist at the top
level, so the gold constraint set was always empty and every triplet metric
collapsed to `None` or `0.0`.

The metric was already doing semantic alignment (`align_by_text` with
SentenceTransformer + Hungarian assignment at threshold 0.75); the matcher was
not the bug.

## Fix

`src/evaluation/metrics.py` adds `normalize_doc_constraints(doc)`. It returns
the existing flat `constraints[]` if present, otherwise it flattens each
`steps[i].constraints.<kind>` list into a top-level view with
`applies_to=step.id` and `type=<kind>`. `evaluate_tier_a_document` now calls
the normalizer on both gold and pred (idempotent on the flat shape).

`extract_constraint_text` also accepts `label` and `expression` as text keys to
match the gold dialect used in `op_firesafety_guideline.json`.

## Findings, not new defaults

- `3M_OEM_SOP` contains zero gold constraints (nested or flat). Coverage and
  AttachmentF1 are correctly `None` for this document under both revisions.
- Headline `Phi` is unchanged at the threshold the paper uses (0.75), because
  pre-fix Coverage was `None` and Phi treats `None` as `0.0`. The post-fix
  value is the explicit `0.0`. No paper number changes at T=0.75.
- AttachmentF1 stays at `0.000` even when Coverage is positive at T=0.50. That
  is a separate signal: the predicted constraint is semantically aligned to a
  gold constraint, but the *step* it points to fails to align to the
  corresponding gold step. This is the real research question the paper needs
  to address, and it is no longer masked by the schema bug.
- Threshold is not changed by this commit. The brief proposed raising
  threshold to 0.85, but 0.85 is stricter than 0.75 and would lower recall
  further. Threshold tuning is a separate experiment.

## Schema accepted by the fix

The normalizer accepts either of:

```json
{"constraints": [{"id": "C1", "text": "...", "applies_to": "S3"}, ...]}
```

or

```json
{
  "steps": [
    {
      "id": "S3",
      "constraints": {
        "guard": [{"id": "C1", "text": "..."}],
        "warning": [{"id": "C2", "label": "..."}],
        "precondition": [],
        "postcondition": [],
        "acceptance_criteria": []
      }
    }
  ]
}
```

Both produce identical metric values when fed through `evaluate_tier_a_document`.

## Unit tests

`tests/evaluation/test_constraint_attachment_metric.py` covers:

- `normalize_doc_constraints` flattens nested step constraints with
  `applies_to=step.id`.
- The normalizer is idempotent on the flat shape and preserves existing link
  keys (`steps`, `attached_to`, `applies_to`, etc.).
- `tier_a_constraints_metrics` returns AttachmentF1 > 0 on a paraphrased pair
  and < 0.1 on an unrelated pair (deterministic; uses an injected stub
  embedder so no model download is needed in CI).
- Repeated calls on the same input return the same dict.

`uv run pytest tests/evaluation/ -v` is green.
